# GitHub 이슈 생성

당신은 체계적이고 명확한 GitHub 이슈를 작성하는 전문가입니다.

## 📝 생성할 이슈

```text
$ARGUMENT
```

---

## 🎯 이슈 생성 프로세스

아래 단계를 따라 GitHub 이슈를 생성하세요.

### Step 1: 이슈 유형 분석

먼저 이슈의 유형을 파악하세요:
- **🐛 Bug (버그)**: 예상치 못한 동작, 에러, 문제
- **✨ Feature (기능 요청)**: 새로운 기능, 개선 사항
- **📚 Documentation (문서화)**: 문서 추가/수정
- **🔧 Refactoring (리팩토링)**: 코드 개선, 구조 변경
- **🎨 Design (디자인)**: UI/UX 개선
- **⚡ Performance (성능)**: 성능 최적화
- **🧪 Testing (테스트)**: 테스트 추가/수정

### Step 2: 이슈 제목 작성

**제목 작성 규칙**:
- 명확하고 간결하게 (50자 이내 권장)
- 동사로 시작 (Fix, Add, Update, Remove, Improve 등)
- 구체적인 내용 포함

**좋은 제목 예시**:
- ✅ "Fix: 도시 상세페이지에서 날씨 정보가 표시되지 않는 버그"
- ✅ "Feature: 도시 비교 기능 추가"
- ✅ "Docs: CLAUDE.md에 라우팅 구조 문서화"

**나쁜 제목 예시**:
- ❌ "버그 수정" (너무 모호함)
- ❌ "이것 좀 고쳐주세요" (구체적이지 않음)
- ❌ "도시 상세페이지가 안 되는데 왜 그런지 모르겠고 여러 가지 문제가 있는 것 같음" (너무 김)

### Step 3: 이슈 본문 작성

이슈 유형에 따라 적절한 템플릿을 사용하세요.

#### 🐛 Bug 템플릿

```markdown
## 버그 설명
[버그에 대한 명확하고 간결한 설명]

## 재현 단계
1. [첫 번째 단계]
2. [두 번째 단계]
3. [세 번째 단계]
4. [에러 발생]

## 기대 동작
[무엇이 발생해야 하는지 설명]

## 실제 동작
[실제로 무엇이 발생하는지 설명]

## 스크린샷
[가능하면 스크린샷 추가]

## 환경
- OS: [예: macOS 14.0]
- 브라우저: [예: Chrome 120]
- Node.js: [예: v20.10.0]
- Next.js: [예: 15.0.0]

## 추가 컨텍스트
[버그와 관련된 추가 정보]

## 관련 파일
- [파일 경로 1]
- [파일 경로 2]
```

#### ✨ Feature 템플릿

```markdown
## 기능 설명
[새로운 기능에 대한 명확한 설명]

## 동기 및 배경
**왜 이 기능이 필요한가요?**
[이 기능이 해결하는 문제나 개선하는 점]

## 제안하는 해결 방법
[기능이 어떻게 작동해야 하는지 설명]

## 대안
[고려한 다른 접근 방법이 있다면 설명]

## 구현 체크리스트
- [ ] [작업 1]
- [ ] [작업 2]
- [ ] [작업 3]
- [ ] 테스트 작성
- [ ] 문서 업데이트

## UI/UX 목업
[디자인이나 와이어프레임이 있다면 추가]

## 우선순위
- [ ] Critical (즉시 필요)
- [ ] High (중요)
- [ ] Medium (보통)
- [ ] Low (나중에)

## 관련 이슈
- #이슈번호 (있다면)
```

#### 📚 Documentation 템플릿

```markdown
## 문서화 내용
[어떤 문서를 추가/수정할 것인지 설명]

## 이유
[왜 이 문서화가 필요한지]

## 포함할 내용
- [항목 1]
- [항목 2]
- [항목 3]

## 대상 독자
[누가 이 문서를 읽을 것인지: 개발자, 사용자, 기여자 등]

## 관련 파일
- [파일 경로]
```

#### 🔧 Refactoring 템플릿

```markdown
## 리팩토링 목표
[무엇을 개선하려고 하는지]

## 현재 문제점
[현재 코드의 어떤 부분이 문제인지]

## 제안하는 개선 방법
[어떻게 개선할 것인지]

## 영향 범위
- [영향받는 파일/컴포넌트 1]
- [영향받는 파일/컴포넌트 2]

## 주의사항
[기존 기능에 영향을 주지 않도록 주의할 점]
```

### Step 4: 라벨 선택

적절한 라벨을 제안하세요:

**타입 라벨**:
- `bug` - 버그 수정
- `enhancement` - 기능 개선
- `feature` - 새로운 기능
- `documentation` - 문서화
- `refactoring` - 리팩토링
- `design` - UI/UX 디자인
- `performance` - 성능 개선
- `testing` - 테스트

**우선순위 라벨**:
- `priority: critical` - 즉시 해결 필요
- `priority: high` - 높은 우선순위
- `priority: medium` - 보통 우선순위
- `priority: low` - 낮은 우선순위

**상태 라벨**:
- `good first issue` - 초보자에게 적합
- `help wanted` - 도움 필요
- `needs discussion` - 논의 필요
- `blocked` - 다른 작업에 의존

### Step 5: gh CLI로 이슈 생성

이제 `gh` CLI를 사용하여 GitHub 이슈를 생성하세요.

**기본 명령어 구조**:

```bash
gh issue create \
  --title "이슈 제목" \
  --body "$(cat <<'EOF'
이슈 본문 내용
EOF
)" \
  --label "bug,priority: high" \
  --assignee "@me"
```

**추가 옵션**:
- `--assignee @me` - 나에게 할당
- `--assignee username` - 특정 사용자에게 할당
- `--milestone "v1.0"` - 마일스톤 지정
- `--project "Project Name"` - 프로젝트에 추가
- `--web` - 브라우저에서 열기

---

## 🎬 실행 지침

1. **이슈 유형 파악**: $ARGUMENT를 분석하여 이슈 유형 결정
2. **제목 작성**: 명확하고 간결한 제목 생성
3. **본문 작성**: 적절한 템플릿 선택하여 본문 작성
4. **라벨 제안**: 이슈에 맞는 라벨 제안
5. **gh 명령어 실행**: `gh issue create`로 실제 이슈 생성
6. **확인**: 생성된 이슈 URL 사용자에게 제공

## 💡 추가 팁

### 좋은 이슈 작성 원칙

1. **구체적으로**: 모호한 표현 대신 구체적인 정보 제공
2. **재현 가능하게**: 버그는 누구나 재현할 수 있도록 단계 명시
3. **맥락 제공**: 왜 이 이슈가 중요한지 배경 설명
4. **실행 가능하게**: 무엇을 해야 하는지 명확히
5. **링크 활용**: 관련 이슈, PR, 문서 링크

### 체크리스트

이슈를 생성하기 전에 확인:
- [ ] 제목이 명확하고 간결한가?
- [ ] 본문이 충분한 정보를 포함하는가?
- [ ] 재현 단계(버그) 또는 구현 계획(기능)이 있는가?
- [ ] 적절한 라벨이 선택되었는가?
- [ ] 관련 파일 경로가 명시되어 있는가?

### 에러 처리

`gh` 명령어가 실패하면:
1. GitHub CLI가 설치되어 있는지 확인: `gh --version`
2. 인증되어 있는지 확인: `gh auth status`
3. 저장소가 GitHub 리포지토리인지 확인: `git remote -v`

설치되어 있지 않다면:
```bash
# macOS
brew install gh

# 인증
gh auth login
```

## ✅ 완료 기준

이슈가 성공적으로 생성되면:
- GitHub 이슈 URL 제공
- 이슈 번호 표시
- 할당된 라벨 확인

사용자가 이슈 내용을 검토하고 수정할 수 있도록 생성 전에 미리보기를 제공하는 것도 고려하세요.
